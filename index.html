Pipeline

How ArchBTL works

Three stages, one result: your AI architecture, visualized.

1

Static Analysis

Tree-sitter parses your code into ASTs across 10+ languages. Call graphs, imports, HTTP endpoints all extracted.

→
2

LLM Enrichment

Code structure sent to BTL Runtime via /v1/chat/completions. The model groups functions into meaningful workflows and generates labels.

→
3

Interactive Graph

Mermaid output parsed, resolved against source locations, rendered as interactive DAG with D3.js and ELK layout.

Powered by

Built entirely on BTL Runtime

Every ArchBTL feature is powered by BTL Runtime endpoints. Here's exactly how.

POST
/v1/chat/completions

Workflow detection, metadata generation, and structure condensation. All requests use the btl-2 model with temperature 0.0 for deterministic output.

POST
/analyze/compare

Multi-model comparison. Sends the same code through BTL Runtime, diffs the workflow graphs, and reports consensus scores with disagreements flagged.

POST
/search

Semantic search across all analyzed workflow nodes. Find "authentication" or "error handling" nodes across your entire codebase in milliseconds.

GET
/v1/models

Health check validates the BTL API key on extension activation. Green light means you're ready to analyze.
